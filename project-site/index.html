Git Submodules in Multi-Repository Projects

This page is my project write-up: what I built, what broke, and how I made submodules work in a clean and repeatable way.

Main Repository External Repository

What I Had To Build

Given Scenario

The main project depends on code maintained in another repository. The challenge was to avoid copy-paste dependency management and keep version tracking reliable for every teammate.

My Approach

I used Git submodules so the main repository points to an exact commit in the external repository. That gave me reproducibility and clear ownership boundaries between repos.

Repository Layout

main-project/
|-- .gitmodules
|-- libs/
|   |-- external-lib/  (submodule)
|-- README.md
|-- SUBMODULE_USAGE.md
|-- PROJECT_REPORT.md
|-- demo.ps1
|-- verify-project.ps1
|-- update-submodule.ps1
|-- project-site/
    |-- index.html
    |-- styles.css
    |-- script.js

How I Implemented It

  1. Created two repositories: main-project and external-lib
  2. Linked external-lib under libs/external-lib as a submodule
  3. Committed .gitmodules and submodule pointer in main-project
  4. Made an upstream change in external-lib and pulled it from main-project
  5. Synchronized URLs and checked status hashes for consistency
  6. Added docs plus verification scripts for easy review

Field Notes (Real Issues I Faced)

While testing local-path submodule updates on Windows, I hit a file transport restriction. I solved it with: git -c protocol.file.allow=always submodule update --remote --merge

After pushing to GitHub, I updated .gitmodules to use the remote URL so other users can clone with --recurse-submodules without local path issues.

Key Commands

Add Submodule

git submodule add <external-repo-url> libs/external-lib

Initialize on Clone

git submodule update --init --recursive

Update Submodule Pointer

git submodule update --remote --merge

Sync URLs

git submodule sync --recursive
